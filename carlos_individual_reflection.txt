I worked primarily on the core “end-to-end” path that ties the project together: shared state + service layer + text interface + system startup, and I also built the evaluation tooling used for the report. The files I focused on were state.py, parking_service.py, text_server.py, and server_main.py, plus the experiment scripts load_test_rpc.py and sensor_spam.py. 

What I implemented: I implemented/organized the shared parking state and the service API around it so both the text interface and RPC layer can call the same logic consistently. I also wired up the server startup so configuration is loaded once and each server component starts with the correct host/port/thread settings. On the evaluation side, I wrote load_test_rpc.py to run the required 30-second RPC load tests across 1/4/8/16 workers and report throughput plus median/p95 latency, and sensor_spam.py to generate 10 updates/sec/lot to stress the async + pub/sub path.

A bug I fixed: I corrected integration issues that show up at runtime (config/entrypoint wiring and request flow consistency) so the system reliably boots and the interfaces behave the same way against the same state.

One design change: I made the system more measurable and report-friendly by keeping business logic centralized (state/service) and separating evaluation tooling from server code. This improved reproducibility: anyone can rerun the baseline vs sensor-stress experiments using the scripts without modifying the server.
